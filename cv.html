Link-building / backlink-monitoring platform (NestJS + MongoDB, pnpm monorepo) — sole architect and primary author

  • Designed the hexagonal architecture from scratch: domain entities with no persistence concerns, invariants enforced in constructors and covered by unit tests requiring no database — instead of embedding business rules in Mongoose schemas, which would have required a live DB just to test a validation rule.
  • Replaced a single fat repository abstraction with narrow, purpose-built read/write ports per domain — let read paths (grid pagination, dashboards, reports) use denormalized DTOs and aggressive query optimization without the write-side entity mapping constraining them, and without touching domain invariants.
  • Built a hybrid event-dispatch system (change-stream watch + polling fallback with lease-based claiming) for eventual consistency between aggregates, avoiding both a full message broker and unsafe direct cross-aggregate writes; per-write-entity in-memory queues preserve ordering while allowing full parallelism across different aggregates.
  • Diagnosed a bulk-persist regression (40s for ~1,150 rows) as a missing index — not an architectural bottleneck — before the team committed to a multi-day queue-splitting refactor; the actual fix cut expected write time to under 1 second.
  • Redesigned a 30K-row scraping bulk-write that was silently hanging 7–21 minutes with no retries into chunked, idempotent upsert batches with per-chunk timeouts and queue-level retry — a deliberate trade of whole-batch atomicity for bounded latency and resilience.
  • Built a manual FIFO semaphore gating access to a small pool of browser contexts (not just a worker-count cap) shared by 14 concurrent scraping workers on one browser-automation sidecar, paired with randomized pointer/scroll pacing per fetch so repeat hits don't share a machine-identical rhythm.
  • Implemented a 3-stage escalating verification pipeline (index probe → static fetch → headless-browser render) as chained BullMQ queues, each with its own concurrency and rate-limit profile.
  • Operate three separate persistent-session browser-automation services (one third-party SEO tool, two accounts on another) rather than one generic scraper — each drives its own headful Chrome sidecar (Xvfb + noVNC) that an operator authenticates once, since each tool has a distinct login flow and anti-automation behavior.
  • Built an adaptive auth-health prober (5-minute cadence when healthy, 30s when not) that auto-pauses/resumes each service's queue on login-state change, plus a continuous ambient mouse/scroll-drift simulation that keeps each session looking human between jobs — the real failure mode here was a silently logged-out or flagged-idle session, not request throughput, so that's what the design defends against.
